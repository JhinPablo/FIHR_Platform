Visión General — ¿Qué es el Proyecto Corte 2?

Sistema clínico completo que extiende el Corte 1 con IA cuantizada, imágenes médicas y frontend profesional

Principio fundacional: El Corte 2 no reemplaza el Corte 1. Lo extiende. El sistema FHIR-Lite con PostgreSQL en Render, doble API-Key, tres roles, paginación, cifrado AES-256 y rate-limiting deben seguir funcionando intactos. Sobre esa base, se agrega: un frontend clínico estético tipo PACS/RIS, microservicios de ML tabular + DL imagen cuantizados para CPU, almacenamiento de imágenes en MinIO S3, un orquestador de inferencias concurrentes, y el flujo obligatorio de firma del RiskReport antes de cerrar un paciente.
🚫
PROHIBIDO terminantemente en el frontend: Streamlit · Gradio · Jupyter Notebooks como interfaz · Panel.io · Dash.

PERMITIDO y recomendado: React 18+ · Vue 3 · Svelte / SvelteKit · Astro · HTML5/CSS/JS moderno · AntiGravity u otros frameworks web innovadores. Librerías de UI: Shadcn/UI, Radix, Mantine, DaisyUI, Chakra. Visualización: Plotly.js, Recharts, D3, Three.js. Animaciones: Framer Motion, GSAP, CSS @keyframes.

Propuestas creativas: glassmorphism clínico · cyberpunk HIS · dark PACS/RIS · visualizaciones 3D de datos médicos · interfaces tipo terminal médico. El diseño debe ser memorable y profesional, comparable a un sistema HIS comercial.

Penalización: usar Streamlit o Gradio descuenta −1.5 pts automáticos del componente frontend y penaliza adicionalmente.

Arquitectura del Sistema

Diagrama de capas, recursos FHIR R4 y esquema de BD extendido

📐 Diagrama de Capas
📋 Recursos FHIR R4 Requeridos
  • Patient — demografía + identification_doc cifrado. active=false = soft-delete.
  • Observation — signos vitales y laboratorio con código LOINC + unidad UCUM. Paginado.
  • Media — imagen JPG/PNG; content.url = URL presignada MinIO; campo modality (FUNDUS, XRAY, DERM…)
  • DiagnosticReport — resultado DL: conclusion + conclusionCode SNOMED/ICD-10 + referencia a ImagingStudy
  • RiskAssessment — resultado ML/DL: prediction[].probability calibrada, qualitativeRisk SNOMED, basis, performer (médico firmante)
  • AuditEvent — log FHIR inmutable: tipo, agente, entidad, fecha, resultado
  • Practitioner — médico con especialidad, ID y referencia a Organization
  • Consent — Habeas Data: status=active, scope patient-privacy, dateTime, IP
🗄️ Esquema de Tablas Nuevas (PostgreSQL)
-- risk_reports: RiskReport firmado por médico CREATE TABLE risk_reports ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), patient_id UUID REFERENCES patients(id), model_type VARCHAR(20), -- 'ML','DL','MULTIMODAL' risk_score NUMERIC(5,4), risk_category VARCHAR(20), is_critical BOOLEAN, prediction_enc BYTEA, -- AES-256 cifrado shap_json JSONB, -- SHAP values o GradCAM URL MinIO fhir_risk_id TEXT, doctor_action VARCHAR(20), -- NULL=sin firmar | 'ACCEPTED' | 'REJECTED' doctor_notes TEXT, -- observaciones obligatorias ≥ 30 chars rejection_reason TEXT, -- obligatorio si REJECTED ≥ 20 chars signed_by UUID REFERENCES users(id), signed_at TIMESTAMPTZ, -- NULL = pendiente de firma médica created_at TIMESTAMPTZ DEFAULT NOW()); -- images: referencias a MinIO (clave cifrada) CREATE TABLE images ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), patient_id UUID REFERENCES patients(id), minio_key BYTEA NOT NULL, -- cifrado pgcrypto modality VARCHAR(50), fhir_media_id TEXT, uploaded_by UUID, created_at TIMESTAMPTZ DEFAULT NOW(), deleted_at TIMESTAMPTZ); -- soft-delete -- inference_queue: cola de inferencias concurrentes CREATE TABLE inference_queue ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), patient_id UUID, model_type VARCHAR(20), status VARCHAR(20) DEFAULT 'PENDING', requested_by UUID, created_at TIMESTAMPTZ DEFAULT NOW(), completed_at TIMESTAMPTZ, result_id UUID, error_msg TEXT); -- audit_log: INSERT-ONLY — jamás UPDATE ni DELETE CREATE TABLE audit_log ( id BIGSERIAL PRIMARY KEY, ts TIMESTAMPTZ DEFAULT NOW(), user_id UUID, role VARCHAR(20), action VARCHAR(80), resource_type VARCHAR(40), resource_id UUID, ip_address INET, result VARCHAR(20), detail JSONB); -- consent: Habeas Data aceptado CREATE TABLE consent ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID REFERENCES users(id), policy_version VARCHAR(20) DEFAULT '1.0', accepted_at TIMESTAMPTZ DEFAULT NOW(), ip_address INET);

Estrategia de Datos — Pacientes Sintéticos desde Datasets

Cada caso del dataset se convierte en un paciente FHIR con observaciones e imagen en MinIO

Concepto clave: No se inventan datos. Cada fila del dataset tabular (ej. PIMA Diabetes) se convierte en un paciente FHIR con sus features como Observations LOINC, y la imagen correspondiente del dataset DL (ej. APTOS 2019) se sube a MinIO como recurso Media vinculado al mismo paciente. El sistema tiene datos clínicos reales desde el primer docker compose up.
🗺️ Mapeo Dataset → FHIR
  • Una fila = un paciente — nombre y datos demográficos generados con Faker('es_CO')
  • Feature tabular → Observation FHIR con código LOINC + unidad UCUM (una Observation por feature)
  • Imagen del dataset DL → subida a MinIO como patients/{id}/image.png → recurso FHIR Media
  • Etiqueta ground_truth → campo oculto en BD para evaluación del modelo (NO visible al paciente en el frontend)
  • Mínimo: 30 pacientes con datos tabulares + 15 con imagen vinculada
  • Script: python scripts/seed_patients.py — ejecutable cuando el sistema está levantado
📦 Datasets Recomendados
  • PIMA Indians Diabetes — UCI ML · 768 casos · 8 features (Glucose, BMI, Insulin, BloodPressure…) · target binario
  • APTOS 2019 — Kaggle · 3662 imágenes fondo de ojo JPG · 5 clases retinopatía (0–4)
  • Heart Disease Cleveland — UCI ML · 303 casos · 13 features · binario
  • RSNA Pneumonia Detection — Kaggle · radiografías torácicas · binario
  • Chest X-Ray NIH — 14 enfermedades pulmonares · PNG
  • El equipo elige la combinación tabular + imagen y justifica la pertinencia clínica en el informe
🌱 Script seed_patients.py — Implementación de Referencia
# scripts/seed_patients.py — Poblar sistema con pacientes sintéticos desde dataset import pandas as pd, pathlib, requests from faker import Faker from minio import Minio faker = Faker('es_CO') df = pd.read_csv('datasets/diabetes.csv') # PIMA Diabetes imgs = sorted(pathlib.Path('datasets/aptos/train').glob('*.png')) mc = Minio("minio:9000", access_key=..., secret_key=..., secure=False) # Mapeo features → LOINC (codificación estándar) LOINC = {'Glucose':'2339-0', 'BloodPressure':'55284-4', 'BMI':'39156-5', 'Insulin':'14749-6', 'Age':'21612-7', 'Pregnancies':'11996-6'} for i, row in df.iterrows(): # [PASO 1] Crear paciente FHIR con ground_truth oculto p = requests.post(f"{API}/fhir/Patient", headers=H, json={ "resourceType":"Patient", "name":faker.name(), "birthDate":str(faker.date_of_birth(minimum_age=20, maximum_age=70)), "identification_doc":faker.numerify('##########'), # se cifrará en backend "ground_truth":int(row['Outcome']) # oculto al rol Paciente }).json() pid = p['id'] # [PASO 2] Crear Observations (una por feature, código LOINC) for col, loinc in LOINC.items(): if col in row and pd.notna(row[col]): requests.post(f"{API}/fhir/Observation", headers=H, json={ "resourceType":"Observation", "subject":pid, "status":"final", "code":{"system":"http://loinc.org", "code":loinc}, "valueQuantity":{"value":float(row[col])}}) # [PASO 3] Subir imagen a MinIO y crear recurso Media FHIR if i < len(imgs): key = f"patients/{pid}/retina.png" mc.fput_object("clinical-images", key, str(imgs[i])) requests.post(f"{API}/fhir/Media", headers=H, json={ "resourceType":"Media", "subject":pid, "status":"completed", "minio_key":key, "modality":"FUNDUS"}) print(f"✅ {len(df)} pacientes sintéticos creados con Observations LOINC e imágenes MinIO")

Microservicios de IA — ML Tabular + DL Imagen + Orquestador

Cuantizados para CPU · Calibrados · SHAP + Grad-CAM · RiskReport FHIR · Concurrencia mínima 4

Restricción de hardware (no negociable): La VPS de deploy tiene solo CPU y RAM — sin GPU, sin CUDA. Los modelos deben estar cuantizados (ONNX o INT8) antes de empaquetar en Docker. Si el contenedor lanza RuntimeError: CUDA not available en la demo, ese componente queda en 0 pts. Prueben con GPU desactivada antes de entregar.
📊 ML Service — Datos Tabulares Obligatorio
  • Algoritmo: XGBoost / Gradient Boosting / Random Forest — elegir uno, justificar con métricas
  • Calibración: CalibratedClassifierCV(method='isotonic', cv=5) — probabilidades clínicamente fiables
  • Exportación: skl2onnx → archivo ml_model.onnx (alternativa: onnxmltools)
  • Inferencia CPU: onnxruntime.InferenceSession(providers=['CPUExecutionProvider'])
  • SHAP: TreeExplainer → valores por feature incluidos en respuesta JSON del endpoint
  • Métricas obligatorias en README: F1-score, AUC-ROC, Precision, Recall — no solo accuracy
  • Endpoints: POST /ml/predict · POST /ml/feedback · GET /ml/version
  • Resultado: guardado como FHIR RiskAssessment + fila en risk_reports (signed_at = NULL)
  • Tiempo máximo CPU: < 3 segundos
🧠 DL Service — Imágenes Obligatorio
  • Arquitectura: EfficientNet-B0 o MobileNetV3 (más ligeras para CPU) fine-tuned sobre el dataset elegido
  • Cuantización PyTorch: torch.quantization.quantize_dynamic(model, {nn.Linear, nn.Conv2d}, dtype=torch.qint8)
  • Alternativa ONNX: torch.onnx.export → onnxruntime CPUExecutionProvider
  • Endpoint: POST /dl/predict — acepta multipart/form-data (imagen) o {"minio_key": "..."}
  • Grad-CAM: mapa de calor generado y guardado en MinIO como gradcam/{patient_id}_{ts}.jpg
  • Resultado: clase predicha, probabilidades por clase, URL Grad-CAM, FHIR DiagnosticReport
  • Endpoints: POST /dl/predict · POST /dl/feedback · GET /dl/version
  • Tiempo máximo CPU: < 15 segundos

🎛️ Orquestador — Concurrencia Mínima 4 Inferencias Simultáneas
  • Patrón: asyncio.Semaphore(MAX_WORKERS) + BackgroundTasks FastAPI
  • MAX_WORKERS=4 — configurable por variable de entorno
  • POST /infer → retorna task_id inmediatamente (no bloquea)
  • GET /infer/{task_id} → PENDING / RUNNING / DONE / ERROR
  • Si hay >4 solicitudes: las restantes esperan en cola con estado PENDING
  • Estado persistido en tabla inference_queue (PostgreSQL)
  • Timeout configurable 120s — si excede → ERROR
  • Frontend: polling cada 3s (o WebSocket) hasta DONE
# orchestrator/main.py import asyncio, httpx, os from fastapi import FastAPI, BackgroundTasks app = FastAPI(title="Inference Orchestrator") sem = asyncio.Semaphore(int(os.getenv("MAX_WORKERS","4"))) async def run_inference(task_id, patient_id, model_type): async with sem: # bloquea si ya hay 4 corriendo await set_status(task_id, "RUNNING") try: url = ML_URL if model_type=="ML" else DL_URL async with httpx.AsyncClient(timeout=120) as c: r = await c.post(f"{url}/predict", json={"patient_id": patient_id}) rid = await save_risk_report(patient_id, r.json()) await set_status(task_id, "DONE", result_id=rid) except Exception as e: await set_status(task_id, "ERROR", error_msg=str(e)) @app.post("/infer") async def request_inference(body: InferReq, bg: BackgroundTasks): tid = await create_queue_entry(body.patient_id, body.model_type) bg.add_task(run_inference, tid, body.patient_id, body.model_type) return {"task_id": tid, "status": "PENDING"}

Flujo Clínico Completo — Login → Cierre Firmado

Cada paso está auditado — el médico no puede cerrar sin firmar el RiskReport

Paso 1
🔐 Login con Doble API-Key + Habeas Data
El médico ingresa X-Access-Key + X-Permission-Key → backend valida ambas → JWT retornado. Si es primer acceso del usuario: modal obligatorio de Habeas Data (Ley 1581/2012 versión 1.0) — no se puede saltar → timestamp + IP guardados en tabla consent + FHIR Consent. Audit log: LOGIN, HABEAS_DATA_ACCEPTED.
Paso 2
👥 Dashboard — Lista de Pacientes Paginada
GET /fhir/Patient?limit=10&offset=0 → tabla tipo PACS con estado del RiskReport. Admin ve todos, Médico ve los asignados, Paciente ve solo el propio. Audit log: LIST_PATIENTS.
Paso 3
📋 Apertura de Ficha Clínica
Clic en paciente → ruta /patients/:id con datos FHIR Patient, Observations en gráficas Plotly.js, imágenes MinIO, historial de RiskReports firmados. Banner rojo si hay pendiente. Audit log: VIEW_PATIENT.
Paso 4
🤖 Solicitud de Análisis al Orquestador
Médico selecciona tipo (ML / DL / Multimodal) → POST /infer → task_id inmediato → frontend polling o WebSocket cada 3s → spinner de progreso. Mínimo 4 análisis simultáneos sin error. Audit log: RUN_INFERENCE.
Paso 5
📊 Visualización del Resultado
risk_score, risk_category, SHAP barras (ML) o Grad-CAM superpuesto (DL). Resultado guardado como FHIR RiskAssessment con signed_at = NULL. Disclaimer IA visible. Si is_critical = true → Paso 6. Audit log: INFERENCE_COMPLETED.
Paso 6 (condicional)
🚨 Gestión de Alerta Crítica
Modal rojo bloqueante — no se puede navegar sin resolver. Email automático Mailhog al médico responsable. Médico revisa SHAP/Grad-CAM y debe seleccionar: Confirmar o Rechazar (justificación ≥ 20 chars si rechaza). Audit log: CRITICAL_ALERT_TRIGGERED, CRITICAL_ALERT_RESOLVED.
Paso 7
✍️ Firma Obligatoria del RiskReport
Médico escribe observaciones clínicas (≥ 30 chars) y selecciona Aceptar o Rechazar. PATCH /fhir/RiskAssessment/{id}/sign → signed_by y signed_at registrados en BD y FHIR. Retroalimentación guardada en model_feedback. Audit log: SIGN_REPORT.
Paso 8
✅ Cierre del Paciente — Bloqueado hasta firma
Backend valida SELECT * FROM risk_reports WHERE patient_id=$1 AND signed_at IS NULL. Si hay pendientes → error 409 "PENDING_SIGNATURE". Solo con firma completa el botón de cerrar se habilita. Audit log: CLOSE_PATIENT.

# backend/routers/patients.py @router.get("/fhir/Patient/{pid}/can-close") async def can_close_patient(pid: str, doctor=Depends(require_medico), db=Depends(get_db)): # Verificar RiskReports sin firma para este paciente pending = await db.fetch( """SELECT id FROM risk_reports WHERE patient_id = $1::uuid AND signed_at IS NULL AND deleted_at IS NULL""", pid) if pending: raise HTTPException( status_code=409, detail={"error":"PENDING_SIGNATURE", "message":"Debe firmar el RiskReport antes de cerrar el paciente", "pending_count":len(pending)}) await log_audit(doctor.id, "CLOSE_PATIENT", "Patient", pid, "SUCCESS") return {"can_close":True, "message":"Paciente puede ser cerrado"}

Seguridad, Privacidad y Cumplimiento Normativo

Anti-DDoS · Cifrado en reposo y tránsito · RBAC · Habeas Data · Normativa colombiana

🛡️ Protección Anti-DDoS
  • Nginx como único punto de entrada — todos los servicios en red Docker interna, sin puertos directamente expuestos al exterior
  • Rate-limiting Nginx: limit_req_zone — 100 req/min/IP en auth; 500 req/min/IP en API; 10 inferencias/min/key
  • Auto-ban: bloqueo de IP tras 10 intentos fallidos de autenticación en 5 minutos; Error 429 con Retry-After
  • Headers HTTP de seguridad: X-Frame-Options: DENY · X-Content-Type-Options: nosniff · Strict-Transport-Security · Content-Security-Policy · Referrer-Policy
  • CORS restrictivo: solo orígenes en ALLOWED_ORIGINS del .env
  • Tamaño máximo de payload: client_max_body_size 10M (imágenes médicas)
  • Timeouts anti-Slowloris: client_body_timeout 10s, send_timeout 10s
🔐 Autenticación y Autorización
  • Doble API-Key (herencia C1) — X-Access-Key + X-Permission-Key en todos los endpoints sin excepción
  • JWT Bearer — para la SPA; expiración 8h; refresh token con rotación; blacklist al logout
  • bcrypt — hashing de contraseñas, work factor ≥ 12
  • RBAC en backend — decoradores @require_medico, @require_admin en cada endpoint FastAPI; el frontend nunca es la única barrera de seguridad
  • Paciente = solo su información — query siempre incluye WHERE owner_id = current_user
  • Política de contraseñas: mínimo 10 chars, mayúscula, número, símbolo
🔒 Cifrado de Datos
  • En tránsito: HTTPS/TLS — self-signed en dev; Let's Encrypt en producción
  • En reposo (PostgreSQL): pgcrypto.encrypt(value, key, 'aes') — campos: identification_doc, medical_summary, risk_prediction, minio_key
  • MinIO SSE: Server-Side Encryption activado para todas las imágenes médicas
  • Secretos en .env: credenciales nunca en código fuente ni en commits
  • Logs seguros: nunca registrar contraseñas, tokens, API-Keys ni PHI en texto plano
  • .gitignore obligatorio: .env, *.onnx, *.pth, carpeta datasets/
📋 Soft-Delete + Audit Log INSERT-ONLY
  • Soft-delete: columna deleted_at TIMESTAMPTZ en todas las entidades; todas las queries filtran WHERE deleted_at IS NULL; DELETE endpoint → UPDATE SET deleted_at = NOW()
  • Restaurar: solo Admin puede restaurar con PATCH /{id}/restore
  • Audit log INSERT-ONLY: jamás UPDATE ni DELETE en la tabla audit_log
  • Acciones mínimas auditadas: LOGIN · LOGOUT · LIST_PATIENTS · VIEW_PATIENT · UPLOAD_IMAGE · RUN_INFERENCE · INFERENCE_COMPLETED · SIGN_REPORT · CRITICAL_ALERT_TRIGGERED · CRITICAL_ALERT_RESOLVED · CREATE_USER · DELETE_USER · HABEAS_DATA_ACCEPTED · CLOSE_PATIENT
  • También como FHIR AuditEvent
  • Admin: ver, filtrar y exportar CSV/JSON
⚖️ Normativa Colombiana y Habeas Data
  • Ley 1581/2012 — Protección de datos personales; dato de salud = dato sensible; consentimiento informado previo y explícito
  • Ley 2015/2020 — Historia Clínica Electrónica Interoperable; unicidad, portabilidad, confidencialidad
  • Resolución 866/2021 (MIAS) — adopción de HL7 FHIR R4 para interoperabilidad semántica en el sistema de salud colombiano
  • Resolución 1995/1999 — retención de HC mínimo 15 años; política de retención documentada en README
  • Derechos ARCO: flujo de solicitud de acceso, rectificación, cancelación y oposición; soft-delete + anonimización
  • Disclaimer IA obligatorio: "Resultado generado por IA de apoyo diagnóstico. No reemplaza criterio médico. Sujeto a revisión clínica."

Interoperabilidad Semántica HL7 FHIR R4

Codificaciones estándar · Terminologías clínicas · Flujo de mensajes

📚 Terminologías y Codificaciones Requeridas
  • LOINC — códigos para Observations (signos vitales, laboratorio). Ej: 2339-0 = Glucosa, 55284-4 = Presión Arterial, 39156-5 = BMI, 14749-6 = Insulina
  • SNOMED CT — diagnósticos en RiskAssessment.qualitativeRisk y DiagnosticReport.conclusionCode. Ej: 73211009 = Diabetes mellitus
  • ICD-10 — código diagnóstico secundario en DiagnosticReport.conclusion
  • UCUM — unidades en Observations. Ej: mg/dL, mmHg, kg/m2
  • RxNorm — medicamentos si se incluye historial farmacológico
  • Sistema de URLs FHIR: http://loinc.org, http://snomed.info/sct, http://hl7.org/fhir/sid/icd-10
🔄 Flujo de Interoperabilidad
  • Dataset → FHIR: script seed mapea features a Observations LOINC → POST al servidor HAPI FHIR R4
  • Imagen → FHIR: upload a MinIO → URL presignada → recurso Media en HAPI FHIR
  • ML Result → FHIR: predicción → RiskAssessment con probability calibrada + qualitativeRisk SNOMED
  • DL Result → FHIR: clasificación → DiagnosticReport con conclusionCode SNOMED/ICD-10 + referencia a Grad-CAM en MinIO
  • Firma médica → FHIR: acción del médico actualiza RiskAssessment.performer y basis
  • Acción de usuario → FHIR: cada acción clínica genera un recurso AuditEvent
  • SMART on FHIR (bono): si implementan autenticación basada en SMART on FHIR con JWT estándar FHIR
📡 Protocolos de Comunicación Requeridos
  • FHIR RESTful API — CRUD con parámetros de búsqueda FHIR (?subject=Patient/{id}, ?date=gt2024, ?code=2339-0)
  • S3 API (MinIO) — upload multipart + URL presignada válida 1h para servir imágenes en el frontend
  • ONNX Runtime CPU — estándar de interoperabilidad de modelos entre plataformas
  • WebSocket — notificaciones tiempo real: estado de inferencia + alertas críticas sin polling
  • SMTP (Mailhog) — alertas críticas al médico por email
  • JWT Bearer (RFC 7519) — autenticación stateless entre frontend y microservicios
  • OpenAPI 3.0 (Swagger) — documentación automática en cada FastAPI (/docs)
  • WADO-RS (Orthanc) — si se implementa el bono PACS: recuperación DICOM por URL
